## ТЗ: Банковский реестр — HTTP API (текстовая версия)

Назначение: REST API для управления банковскими счетами и переводами между ними. Основной акцент — корректность работы под конкурентной нагрузкой: ACID-транзакции, безопасные блокировки, обработка параллельных запросов средствами горутин.

Общие технические требования. Приложение представляет собой HTTP-сервер на стандартном net/http (или с лёгким роутером — chi, gorilla/mux). Все денежные суммы хранятся и передаются как целые числа в копейках (int64), использование float для денег запрещено во избежание ошибок округления. Все запросы и ответы — в формате JSON. Сервер должен корректно обрабатывать конкурентные запросы: одновременные переводы с одного счёта не должны приводить к потере обновлений или отрицательному балансу. Приложение обязано проходить go test -race без предупреждений. Реализуется graceful shutdown: при получении сигнала завершения сервер перестаёт принимать новые соединения, дожидается завершения текущих запросов через context и только потом останавливается.

Доменные сущности. Счёт (Account) содержит уникальный идентификатор, имя владельца и баланс в копейках. Транзакция (Transaction) содержит идентификатор, счёт-отправитель, счёт-получатель, сумму, статус (ожидает, выполнена, отклонена) и время создания. Операции пополнения и снятия можно моделировать как транзакции с пустым полем отправителя или получателя соответственно.

Требования к переводу (ключевая операция). Перевод должен обладать всеми свойствами ACID. Атомарность: либо средства и списаны, и зачислены, либо операция полностью откатывается; промежуточное состояние, где деньги списаны, но не зачислены, недопустимо даже при сбое посередине. Согласованность: суммарный баланс всех счетов до и после перевода неизменен; баланс не может стать отрицательным. Изоляция: два параллельных перевода с одного счёта не должны терять обновления — итоговый баланс должен учитывать оба списания. Долговечность: после успешного ответа сервера перевод сохранён и переживает перезапуск приложения. Для обеспечения изоляции используются блокировки на уровне отдельных счетов. Чтобы исключить взаимную блокировку при встречных переводах (A→B и одновременно B→A), счета захватываются всегда в детерминированном порядке — например, по возрастанию идентификатора.

Конкурентность. Сервер по своей природе обрабатывает каждый HTTP-запрос в отдельной горутине, поэтому доступ к общему состоянию (счетам) должен быть защищён. При реализации эндпоинта массовых переводов используется пул воркеров: задания раздаются через канал, завершение всех заданий отслеживается через sync.WaitGroup, отмена и таймауты — через context.Context. Счётчики успешных и неуспешных операций ведутся через sync/atomic.

Хранилище. Доступ к данным организован через интерфейс репозитория, объявленный на стороне бизнес-логики. Реализуется минимум одна персистентная версия (например, на PostgreSQL или SQLite), где атомарность перевода обеспечивается транзакцией базы данных (BEGIN, COMMIT, ROLLBACK). Желательна также in-memory реализация с защитой через мьютексы — для тестов и демонстрации второго подхода к атомарности. Слои приложения разделены: обработчики HTTP принимают и валидируют запрос и формируют ответ; сервис содержит бизнес-логику переводов и блокировок; репозиторий отвечает за хранение. Зависимости направлены строго от обработчиков к сервису и далее к репозиторию.

Обработка ошибок и валидация. Каждый эндпоинт валидирует входные данные: суммы должны быть положительными, счета должны существовать, при снятии и переводе проверяется достаточность средств. Ошибки возвращаются с корректными HTTP-статусами: 400 при некорректном запросе, 404 при отсутствии счёта, 409 или 422 при недостатке средств, 500 при внутренней ошибке. Тело ответа с ошибкой содержит понятное сообщение в формате JSON. Внутренние детали ошибок (стек, SQL) наружу не отдаются.

Тестирование. Ключевой тест корректности: создаётся фиксированное число счетов с известным суммарным балансом, затем конкурентно (через множество горутин) выполняется большое количество случайных переводов между ними; после завершения проверяется, что суммарный баланс не изменился, ни один счёт не ушёл в минус, и детектор гонок не нашёл проблем. Дополнительно покрываются тестами отдельные сценарии: перевод при недостатке средств, перевод на несуществующий счёт, корректность HTTP-статусов.

Критерии готовности. Все обязательные эндпоинты реализованы и валидируют ввод. Команды go test ./... и go test -race ./... проходят успешно. Ключевой конкурентный тест стабильно зелёный. Взаимная блокировка при встречных переводах невозможна. Реализован graceful shutdown. Код проходит golangci-lint без замечаний. Рекомендуемый порядок реализации. Сначала — доменная модель и in-memory репозиторий с базовыми операциями создания счёта, пополнения и снятия. Затем — эндпоинт перевода с блокировками на уровне счетов и упорядоченным захватом. После этого — конкурентный тест на сохранность суммы и проверка race detector. Далее — пул воркеров для массовых переводов с горутинами, каналами, WaitGroup и context. Затем — персистентный репозиторий с транзакциями базы данных для обеспечения долговечности. В последнюю очередь — graceful shutdown, единообразная обработка ошибок и документация. Проект остаётся работоспособным уже после третьего шага.
